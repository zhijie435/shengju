# 圣举人才网用户端 — 2000 人同时在线性能优化方案

> 分析日期：2026-06-10  
> 目标：稳定支撑 **2000 人同时在线**（`/user/login.html`、`/index.html`、`/user/profile.html` 等用户端页面）  
> 覆盖场景：登录 / 注册 / 简历浏览 / 岗位浏览 / 消息中心 / 报名 / 测评查看（读多写少）

---

## 一、并发模型说明

"2000 人同时在线"与"2000 并发请求"是不同概念，需先建立合理的流量模型：

| 指标 | 估算值 | 说明 |
|------|--------|------|
| 同时在线 | 2000 人 | 已登录并活跃使用（非空闲 Tab） |
| 活跃请求率 | ~15% | 任一秒内约 300 人正在触发请求 |
| 平均峰值 QPS | **200–350 req/s** | 正常浏览、翻页、消息轮询 |
| 登录高峰 QPS | **50–100 req/s** | 考试报名截止前半小时集中登录 |
| 报名写入 QPS | **20–60 req/s** | 报名表单提交（带文件上传） |
| 消息轮询 QPS | **60–120 req/s** | profile 消息 Tab `_t=` 强刷（每人 30s 一次） |
| 数据库读 QPS | **500–1000** | 含 1 次请求多次 SQL 放大（当前平均 3–5 倍） |
| 数据库写 QPS | **30–80** | 报名、简历更新、已读标记 |

**结论：当前单进程架构在不做任何优化的情况下，约能稳定承载 300–500 人同时在线**（主要受数据库连接池和 bcrypt CPU 限制）。达到 2000 人需系统性改造。

## 二、核心性能瓶颈（代码层实证）

### 🔴 P0 — 直接影响用户可用性

#### 1. 登录路径 CPU 与 DB 双重过载

**现状代码路径**（`backend/routes/auth.js`）：

```
POST /api/v1/auth/login
  → 手机号查询：SELECT * FROM qms_users WHERE phone IN (?, ?)   ← SELECT *，可能返回多账号
  → 对每个账号：bcrypt.compare(password, hash, rounds=10)       ← 每次 ~80ms CPU 阻塞
  → assertLoginPortalAllowed → EnterpriseModel.findByUserId     ← 额外 DB 查询
  → enrichUserIdCardFromCoopApplication                         ← SELECT * FROM compat_cooperation_applications
  → 失败时 fallback: trySyncSjJobseekerByPhone                  ← 可能再次 DB 查询
```

**问题**：50 人同时登录 → 50 × 80ms bcrypt → Node.js 单线程事件循环被 bcrypt 阻塞约 4 秒，其他所有请求排队。

**估算影响**：单进程下登录并发超过 20 人即出现明显延迟。

---

#### 2. `selectQmsUserRowForProfile` 每次 5 次 SQL

**现状代码**（`backend/routes/talentSiteCompat.js`）：

```javascript
// 1 次基础查询
await pool.execute('SELECT id, username, email, real_name, phone, role, education FROM qms_users WHERE id = ?', [userId]);

// 4 次动态列探测（列不存在则跳过）

  await pool.execute(`SELECT \`${col}\` AS _v FROM qms_users WHERE id = ? LIMIT 1`, [userId]);
}
```

**问题**：个人中心首屏必打，2000 人每次打开 profile → **10000 次 DB 查询**（5次/人）。

#### 3. `getCompatNotifications` 消息中心重查询

```
GET /api/v1/notifications?_t=1749521234  （每次强刷，无缓存）
  → 获取手机关联账号 ID（SELECT）
  → 查 compat_cooperation_applications（LEFT JOIN projects + announcements）
  → 查 exam_admit_cards（2次 SQL）
  → 查 exam_score_notifications（SELECT）
  → 循环内逐条查公告标题 announcementTitle(id)  ← N+1 问题！
  合计：3～5 条 SQL + N 次公告查询
```

**问题**：消息 Tab 每 30 秒强刷一次（`_t=` 参数绕过缓存），300 人同时看消息 → **每 30s 触发 1500～3000 次 SQL**。

#### 4. `putCompatUserProfile` 逐字段 UPDATE


**问题**：单次保存简历基本信息占 7 个连接槽，高峰期直接堵塞连接池。

#### 5. `optionalAuthenticate` 不走 LRU 缓存

**`backend/middleware/auth.js:387`**：

```javascript

// optionalAuthenticate 直接查库（无缓存！）
```

`/jobs`、`/announcements`、`/public/site-settings` 等公开浏览接口都用 `optionalAuthenticate`，所有带 token 的用户请求都多一次 DB 往返。

**`backend/server.js`**：

## 四、潜在架构级风险

### 风险 A：单进程单线程是根本瓶颈

Node.js 单进程，任何一次 bcrypt 计算（80ms）或大 JSON 序列化（>1MB，~50ms）都会阻塞所有并发请求。2000 人同时在线时，峰值 300 QPS × 平均 20ms/请求 = 事件循环利用率 **6000%**（远超 100%），必然积压。

**解决方向**：PM2 cluster 多进程（按 CPU 核数扩展）+ Redis 共享状态。

---

### 风险 B：数据库连接池在高峰必然耗尽

| 场景 | 连接需求 | 可用连接 |
|------|---------|---------|
| 正常浏览（200 QPS × 平均 3 SQL × 5ms） | **~30 个** | 50 |
| 登录高峰（50 登录 × 5 SQL + bcrypt 阻塞期间请求堆积） | **~50+ 个** | 50（耗尽） |
| 报名峰期（60 QPS × 4 SQL） | **~50 个** | 50（耗尽） |
| 管理员触发名单导入（串行 N×3 SQL） | **1 个（长期占用）** | — |

连接池耗尽后新请求进入等待队列（`queueLimit=500`），用户感知为 3–10 秒超时。

---

### 风险 C：`talentSiteCompat.js` 8959 行单体模块

- 启动时完整 `require` 进内存，所有函数共享同一个闭包作用域
- 内存泄漏风险高（`talentPoolJoinedUserIds` Set 持续增长）
- 任意一处未捕获异常可能影响全部 compat 接口

---

### 风险 D：静态资源全部由 Node.js 服务

```javascript
// server.js — Node.js 直接提供静态文件
app.use(express.static(legacyShengjuDir, { maxAge: '1d' }));
```

2000 人同时访问，静态资源（JS/CSS/图片）的 HTTP 请求也进入 Node.js 的 libuv 队列。高峰期静态资源请求与 API 请求竞争同一进程资源。

---

### 风险 E：WebSocket 考试监控在同机部署时抢资源

人才网用户端与考试 WebSocket 服务共用同一个 Node.js 进程。考试高峰（50+ 人同时作答时 WS 心跳广播）会与用户端 HTTP 请求争 CPU 和连接池。

#### 2.2 合并 putCompatUserProfile 为单次 UPDATE

```javascript
// 修改前：逐字段 UPDATE，最多 7 次
// 修改后：动态构建单条 UPDATE

async function putCompatUserProfile(userId, updates) {
  const allowedFields = ['real_name', 'gender', 'birth_date', 'location', 'avatar_url', 'education'];
  const setClauses = [];
  const values = [];

  for (const field of allowedFields) {
    if (updates[field] !== undefined) {
      setClauses.push(`\`${field}\` = ?`);
      values.push(updates[field]);
    }
  }

  if (setClauses.length === 0) return;

  values.push(userId);
  await pool.execute(
    `UPDATE qms_users SET ${setClauses.join(', ')}, updated_at = NOW() WHERE id = ?`,
    values
  );
}
```

#### 2.6 短信验证码改 Redis 存储

```javascript
// backend/routes/auth.js
const redis = require('../services/redisClient');

// 存储验证码（替代 Map）
async function setSmsCode(phone, code) {
  await redis.setex(`sms:${phone}`, 300, code);  // 5 分钟过期
}

// 验证码校验
async function verifySmsCode(phone, inputCode) {
  const stored = await redis.get(`sms:${phone}`);
  if (!stored || stored !== inputCode) return false;
  await redis.del(`sms:${phone}`);  // 用完即删
  return true;
}

// 发送冷却（60 秒内不可重发）
async function assertSmsCooldown(phone) {
  const key = `sms_cooldown:${phone}`;
  const existing = await redis.get(key);
  if (existing) throw new Error('请等待 60 秒后再获取验证码');
  await redis.setex(key, 60, '1');
}
```

### 阶段三：中期优化（1–2 个月，支撑 2000 人目标）

#### 3.1 PM2 Cluster 多进程（最关键的架构改变）

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'sjrcw-backend',
    script: 'backend/server.js',
    instances: 'max',           // 按 CPU 核数，推荐 4 核 ECS 启动 3 个（留 1 核给 MySQL）
    exec_mode: 'cluster',
    max_memory_restart: '1.5G',
    env_production: {
      NODE_ENV: 'production',
      PORT: 3000,
    },
    // 进程崩溃自动重启
    restart_delay: 2000,
    max_restarts: 10,
  }]

```

**注意**：启用 cluster 后必须同步改造：

```javascript
// backend/routes/talentSiteCompat.js

// 岗位列表：改为真分页 + 结果缓存
const jobsListCache = new LRU({ max: 100, ttl: 60_000 });  // 60s 缓存

router.get('/jobs', optionalAuthenticate, async (req, res) => {
  const { page = 1, pageSize = 20, enterpriseId, status } = req.query;
  const cacheKey = `jobs:${enterpriseId || 'all'}:${status || 'open'}:${page}:${pageSize}`;

  const cached = jobsListCache.get(cacheKey);
  if (cached) return res.json(cached);

  const offset = (parseInt(page) - 1) * parseInt(pageSize);
  const [rows] = await pool.execute(
    `SELECT j.id, j.title, j.salary_range, j.location, j.created_at, e.name AS enterprise_name
     FROM compat_enterprise_jobs j
     LEFT JOIN enterprises e ON e.id = j.enterprise_id
     WHERE j.status = ? ${enterpriseId ? 'AND j.enterprise_id = ?' : ''}
     ORDER BY j.created_at DESC
     LIMIT ? OFFSET ?`,
    [status || 'open', ...(enterpriseId ? [enterpriseId] : []), parseInt(pageSize), offset]
  );

  const result = { data: rows, page: parseInt(page), pageSize: parseInt(pageSize) };
  jobsListCache.set(cacheKey, result);
  res.json(result);
```

### 8.2 ECS 配置详解

**推荐型号**：`ecs.c7.xlarge`（4 vCPU / 16GB / 内网带宽 4Gbps）

```
为什么选 c7（计算优化）而不是 g7（通用型）？
- Node.js bcrypt/JSON 序列化是 CPU 密集型
- PM2 cluster 3 个 Node.js 进程 + Nginx，共需 4 个 vCPU
- 16GB RAM：Node.js 进程各约 512MB-1GB，留余量给系统和文件缓存
```

**ECS 系统盘**：100GB SSD（ESSD PL1）— 存放代码、日志、上传文件

**ECS 扩展数据盘**：200GB SSD — 挂载为 `/data`，存放用户上传视频/Word/图片

**操作系统**：Alibaba Cloud Linux 3（与 CentOS 8 兼容，内核优化更好）

### 8.3 RDS MySQL 配置详解

**推荐规格**：`mysql.n4.large.1`（4 vCPU / 16GB）高可用版（主备）

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `innodb_buffer_pool_size` | 10GB | 热数据留在内存，减少磁盘 IO |
| `max_connections` | 500 | Node.js pool 50 × 3 进程 = 150 + 余量 |
| `innodb_io_capacity` | 4000 | SSD 磁盘 IOPS 充分利用 |
| `slow_query_log` | ON, 阈值 1s | 捕获慢查询 |
| `innodb_flush_log_at_trx_commit` | 2 | 性能/安全平衡（读写密集可设为 2） |

**存储**：200GB ESSD PL1（3000 IOPS），开启自动扩容（上限 1TB）

**备份策略**：每日全量备份（凌晨 2:00）+ Binlog 实时备份，保留 7 天

Redis 客户端已在 `services/redisClient.js` 中创建，但 `auth.js` 中的短信验证码存储和发送频控窗口**均未切换到 Redis**。

**影响**：PM2 cluster 启动 3 个进程后，用户在进程 A 发送验证码，路由到进程 B 验证，必然失败（验证码在进程 A 内存中）。短信登录功能在多进程下**完全不可用**。

**修复**（`auth.js` 接入 redisClient）：

```javascript

// setSmsCode 改 Redis
async function setSmsCode(phone, code) {
  await redisClient.setex(`sms:${phone}`, 300, code);  // 5 分钟
  await redisClient.setex(`sms_cooldown:${phone}`, 60, '1');  // 60s 冷却
}

// getSmsCode 改 Redis
async function getSmsCode(phone) {
  return await redisClient.get(`sms:${phone}`);
}

// 消费验证码（用完即删）
async function verifyAndConsumeSmsCode(phone, code) {
  const stored = await redisClient.get(`sms:${phone}`);
  if (!stored || stored !== String(code)) return { ok: false, message: '验证码错误或已过期' };
  await redisClient.del(`sms:${phone}`);
  return { ok: true };
}

// 发送频控改 Redis（跨进程共享）
async function assertSmsSendAllowance(norm) {
  const cooldown = await redisClient.get(`sms_cooldown:${norm}`);
  if (cooldown) return { ok: false, message: '请等待 60 秒后再获取验证码' };
  // 小时级次数限制
  const key = `sms_hourly:${norm}`;
  const count = parseInt(await redisClient.get(key) || '0', 10);
  const maxPerHour = parseInt(process.env.SMS_MAX_SENDS_PER_PHONE_PER_HOUR || '4', 10);
  if (count >= maxPerHour) return { ok: false, message: '该号码本小时获取验证码次数已达上限' };
  return { ok: true };
}
```

### 🟠 新隐患 4：`findByIdCard` 函数包裹索引列，无法走索引

**当前代码**（`userModel.js:275`）：

### 🟡 新隐患 5：`sync-portal-passwords` 仍阻塞事件循环

**当前代码**（`talentSiteCompat.js:4447–4469`）：

```javascript
sql += ' ORDER BY ... LIMIT 5000';
const [rows] = await pool.execute(sql, params);
// ...

  await UserModel.syncPortalPasswordFromIdCard(uid, idc);  // 串行 bcrypt(10) × 5000
```

最坏情况：5000 条 × 每次 bcrypt 80ms = **400 秒**阻塞，期间 Node.js 事件循环完全停滞，所有用户请求排队。虽然是低频管理操作，但一旦触发影响所有 2000 在线用户。

**修复**：增加速率控制，分批异步处理：

```javascript
// 每批 10 个用户，批间 yield 事件循环
async function syncPasswordsBatch(rows) {
  for (let i = 0; i < rows.length; i++) {
    const r = rows[i];
    await UserModel.syncPortalPasswordFromIdCard(r.uid, r.idc);
    if (i % 10 === 9) {
      await new Promise(resolve => setImmediate(resolve));  // 让出事件循环
    }
  }
}
```

或更好的方案：将此接口改为后台异步任务，立即返回 `{ taskId }` 响应，前端轮询进度。

### 🟡 新隐患 6：`notificationListCache` 未设上限，可能内存泄漏

**当前代码**（`talentSiteCompat.js:6779`）：

```javascript
const notificationListCache = new Map(); // 无大小上限！
```

普通 `Map` 不会自动淘汰。2000 人同时在线时，Map 持续增长（每人一条 entry）。数据不大（每条约 10-50KB），但：
- 进程重启后所有缓存丢失（不影响功能，只是首次全 miss）
- 长期运行 Map 条目累积（用户离线后 entry 仍存在）

**修复**：改为 LRU 缓存：

```javascript
const { LRUCache } = require('lru-cache');
const notificationListCache = new LRUCache({
  max: 3000,        // 最多缓存 3000 个用户（覆盖 2000 人在线 + 余量）
  ttl: 30_000,      // 30 秒 TTL
});
```
