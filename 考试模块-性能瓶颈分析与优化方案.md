# 考试模块性能瓶颈分析与优化方案

> 分析日期：2026-06-09  
> 目标：稳定支撑 **50 人并发在线考试**  
> 范围：考生端作答流程 + 管理端大数据加载

---

## 一、当前并发承载能力估算

| 场景 | 估计并发上限 | 主要限制因素 |
|------|-------------|-------------|
| 纯粹在线作答（无监控） | ~80–120 人 | 自动保存 DB 热路径 |
| 开启摄像监控 + 截图 | ~30–50 人 | WS 内存缓存仅 200 条、单进程 CPU |
| 同时交卷高峰 | ~20–30 人无感知延迟 | 汇总并发 + 连接池争抢 |
| 管理员同时打开汇总页 | 1–2 人不卡顿 | `ensureExamSummaries` 同步阻塞 |

**结论**：当前现状可稳定支撑约 **20–35 人**并发（开监控时），完成 P0 修复后可达 **50–60 人**。

---

## 二、核心性能瓶颈

### 🔴 P0 — 直接影响考生体验

#### 1. 自动保存热路径过重（每次 3–4 次 DB 往返）

每次考生保存答案的调用链：

```
POST /api/exam-answers (autosave)
  → ExamSessionModel.findById()        ← 三表 JOIN，仅用于验证身份
  → ExamAnswerModel.upsert()
      → 若 uk_session_subq 索引未建：SELECT + UPDATE  （2 次往返）
      → 若索引已建：INSERT ... ON DUPLICATE KEY UPDATE （1 次往返）
```

**影响**：50 人 × 每 30 秒保存 ≈ 持续 5–8 QPS，每次 3–4 次查询 → 实际 DB 压力 **15–30 QPS**，交卷高峰时翻倍。

#### 2. `uk_session_subq` 唯一索引可能未建

`examAnswerModel.js` 代码依赖此索引实现高效 upsert，但迁移脚本中**未找到对应 `ALTER TABLE`**。若未建，每次 autosave 退化为 SELECT + UPDATE 两次往返。

#### 3. 开考时 `getExamPaperComplete` N+1 查询

```javascript
// backend/models/examPaperModel.js
const majorQuestions = await this.getMajorQuestionsByPaperId(id);
for (const majorQ of majorQuestions) {
  majorQ.subQuestions = await this.getSubQuestionsByMajorQuestionId(majorQ.id);  // N 次！
}
```

10 大题试卷 = **11 次 DB 往返**，50 人同时点「开始考试」→ **550+ 次瞬间并发查询**。

#### 4. 机房 NAT 同 IP 限流必然触发

```javascript
// backend/server.js
windowMs: 15 * 60 * 1000,
max: 600  // 每 IP 每 15 分钟 600 次
```

机房同出口 IP 下，50 人 autosave 每 30s 一次 = 每 15 分钟 **1500 次**，必然触发 429 限流导致答案保存失败。

---

### 🟠 P1 — 监控与交卷高峰

#### 5. 截图内存缓存容量不足

```javascript
// backend/services/examWebSocket.js
const screenshotCache = new LRU({ max: 200 });  // 全局仅 200 条
```

50 人 × 每 2 分钟截图 1 次 = 25 分钟内产生 **625 张**，全部超出容量，监考端拉取大量 404。

#### 6. 交卷高峰 DB 风暴

50 人同时交卷 → 触发 50 个 `generateSummary` 异步任务并发，每个任务 **6–8 次 SQL** → 瞬间 **300–400 次查询**争抢 50 个连接池，排队延迟严重。

#### 7. `findOrCreate` 入场三次往返

```javascript
// backend/models/examSessionModel.js
const [rows] = await pool.execute('SELECT * FROM exam_sessions WHERE ...');
if (rows[0]) return rows[0];
const id = await this.create({ ... });                  // 第 2 次
const [newRows] = await pool.execute('SELECT * WHERE id = ?', [id]);  // 第 3 次
return newRows[0];
```

#### 8. WS 心跳广播无节流

50 考生 × 每 15–30s 心跳 → 每次心跳同步遍历监考端 Set 广播，无聚合无背压，与 HTTP autosave 争 CPU。

---

### 🟡 P2 — 管理端大数据量加载

#### 9. 成绩汇总列表同步阻塞（最危险）

```javascript
// backend/services/examSummaryService.js
for (const s of sessions) {
  if (!existing.has(sid)) {
    await this.generateSummary(sid);  // 逐个同步！每个 6–8 次 SQL
  }
}
```

管理员打开「成绩汇总」→ 对所有未生成汇总的交卷会话**同步逐个生成** → 200 人考试可能导致**页面超时或后端卡死数分钟**。

#### 10. 统计接口全量加载无分页

```javascript
// backend/services/examSummaryService.js
const summaries = await ExamSummaryModel.findByExamId(examId);  // 无 LIMIT！
```

千人考试时加载全部 summary 行（含多个大 JSON 字段），在 Node 内存中聚合，响应体轻松超 10MB。

#### 11. 阅卷答案一次性全量拉取

阅卷端拉取某考试全部主观题答案，包含 `content_html`、`full_content` 等大字段，响应体可达数十 MB，序列化时长期阻塞事件循环。

#### 12. 监控视频分片全表扫描

`listLatestByExam` 拉取全部 video_chunks → JS 层取最新，chunks 越积越多查询越慢。

#### 13. 监控存档 ZIP 全量读盘进内存

`GET /archive/exam/:examId` 一次性读所有录像文件进内存再打包，极易 OOM。

## 三、缺失索引清单

| 缺失项 | 影响查询 |
|--------|---------|
| `exam_answers (session_id, sub_question_id)` 唯一索引 | autosave upsert 优化 |
| `exam_sessions (exam_id, status)` 复合索引 | 交卷列表、汇总扫描 |
| `exam_video_chunks (session_id, chunk_type, created_at DESC)` | latest 查询全表扫 |
| `exam_monitor_events (session_id, occurred_at)` | 按会话查事件 |

## 四、优化方案

### 方案一：立即修复（影响基本可用性）

#### 1.1 补建缺失唯一索引

```sql
-- 确认并添加 exam_answers 唯一索引（如不存在）
ALTER TABLE exam_answers
  ADD UNIQUE INDEX uk_session_subq (session_id, sub_question_id);

-- exam_sessions 补充复合索引
ALTER TABLE exam_sessions
  ADD INDEX idx_exam_status (exam_id, status);

-- 视频分片最新查询优化
ALTER TABLE exam_video_chunks
  ADD INDEX idx_session_type_time (session_id, chunk_type, created_at DESC);

-- 监控事件查询优化
ALTER TABLE exam_monitor_events
  ADD INDEX idx_session_time (session_id, occurred_at);
```

#### 1.2 修复机房 IP 限流

```javascript
// backend/server.js
// 考试核心接口改为按用户 ID 限流，提高阈值
const examLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 3000,
  keyGenerator: (req) => req.user?.id || req.ip,  // 按用户 ID 而非 IP
  message: { code: 429, message: '请求过于频繁，请稍后重试' },
});
app.use('/api/exam-answers', examLimiter);
app.use('/api/exam-sessions', examLimiter);
```

#### 1.3 autosave 改用轻量 session 校验

```javascript
// backend/routes/examAnswers.js
// 现在：ExamSessionModel.findById() → 三表 JOIN
// 改为：只查身份相关字段，不 JOIN 用户表和考试表
const [[session]] = await pool.execute(
  `SELECT id, user_id, exam_id, status
   FROM exam_sessions
   WHERE id = ? AND user_id = ? AND status = 'in_progress'`,
  [sessionId, req.user.id]
);
if (!session) return res.status(403).json({ message: '会话无效或已结束' });
```

### 方案二：短期优化（1–2 周，50 人并发稳定）

#### 2.1 试卷加载 N+1 → 单次 JOIN

```javascript
// backend/models/examPaperModel.js
static async getExamPaperComplete(paperId) {
  const [rows] = await pool.execute(`
    SELECT
      mq.id AS mq_id, mq.content AS mq_content, mq.sort_order AS mq_sort,
      sq.id AS sq_id, sq.content, sq.answer_type, sq.score,
      sq.standard_answer, sq.options, sq.sort_order AS sq_sort
    FROM exam_paper_major_questions mq
    LEFT JOIN exam_paper_sub_questions sq ON sq.major_question_id = mq.id
    WHERE mq.paper_id = ?
    ORDER BY mq.sort_order, sq.sort_order
  `, [paperId]);
  return assemblePaperStructure(rows);  // JS 层组装嵌套结构
}
```

#### 2.2 试卷内容进程级缓存（考试期间试卷不变）

```javascript
// backend/models/examPaperModel.js
const _paperCache = new Map();

static async getExamPaperComplete(paperId) {
  if (_paperCache.has(paperId)) return _paperCache.get(paperId);
  const paper = await fetchPaperFromDB(paperId);
  _paperCache.set(paperId, paper);
  // 30 分钟后自动失效
  setTimeout(() => _paperCache.delete(paperId), 30 * 60 * 1000);
  return paper;
}

// 试卷更新时主动清除缓存
static invalidateCache(paperId) {
  _paperCache.delete(paperId);
}
```

#### 2.3 截图改存磁盘，不占堆内存

```javascript
// backend/services/examWebSocket.js
const screenshotDir = path.join(uploadsDir, 'screenshots');

async function storeScreenshot(sessionId, seq, base64Data) {
  const dir = path.join(screenshotDir, sessionId);
  await fs.mkdir(dir, { recursive: true });
  const filePath = path.join(dir, `${seq}.jpg`);
  await fs.writeFile(filePath, Buffer.from(base64Data, 'base64'));
  return filePath;
}

// HTTP 接口直接返回文件流，不走内存
router.get('/screenshots/:sessionId/:seq', (req, res) => {
  const filePath = path.join(screenshotDir, req.params.sessionId, `${req.params.seq}.jpg`);
  res.sendFile(filePath);
});
```

#### 2.4 批量答案改单条 multi-row UPSERT

```javascript
// backend/routes/examAnswers.js（批量保存接口）
router.post('/batch', auth, async (req, res) => {
  const { sessionId, answers } = req.body;
  if (!answers?.length) return res.json({ success: true });

  const placeholders = answers.map(() => '(?,?,?,?)').join(',');
  const values = answers.flatMap(a => [sessionId, a.subQuestionId, a.answerData, a.timeCost]);

  await pool.execute(`
    INSERT INTO exam_answers (session_id, sub_question_id, answer_data, time_cost)
    VALUES ${placeholders}
    ON DUPLICATE KEY UPDATE
      answer_data = VALUES(answer_data),
      time_cost = VALUES(time_cost),
      updated_at = NOW()
  `, values);

  res.json({ success: true });
});
```

#### 2.5 WS 心跳聚合推送（5 秒一批）

```javascript
// backend/services/examWebSocket.js
let heartbeatBuffer = {};

function bufferHeartbeat(sessionId, data) {
  heartbeatBuffer[sessionId] = { ...data, ts: Date.now() };
}

setInterval(() => {
  if (Object.keys(heartbeatBuffer).length === 0) return;
  broadcastToMonitors({ type: 'heartbeat_batch', data: heartbeatBuffer });
  heartbeatBuffer = {};
}, 5000);
```

### 方案三：中期优化（管理端大数据，1 个月）

#### 3.1 汇总生成改为异步，不阻塞列表请求

```javascript
// backend/routes/examSummaries.js
router.get('/exam/:examId', auth, async (req, res) => {
  const { page = 1, pageSize = 20 } = req.query;

  // 1. 直接返回已有数据，不等待生成
  const summaries = await ExamSummaryModel.findByExamId(examId, { page, pageSize });
  const pendingCount = await ExamSummaryModel.countPendingSessions(examId);

  // 2. 有待生成的则后台异步处理，不阻塞响应
  if (pendingCount > 0) {
    setImmediate(() => ExamSummaryService.batchGenerateAsync(examId));
  }

  res.json({ summaries, pendingCount, generating: pendingCount > 0 });
});
```

#### 3.2 统计接口改为 SQL 聚合，不全量加载

```sql
-- backend/models/examSummaryModel.js → getExamStatistics
SELECT
  COUNT(*)                                           AS total_count,
  ROUND(AVG(total_score), 2)                         AS avg_score,
  MAX(total_score)                                   AS max_score,
  MIN(total_score)                                   AS min_score,
  COUNT(CASE WHEN total_score >= ? THEN 1 END)       AS pass_count,
  ROUND(AVG(answer_rate), 2)                         AS avg_answer_rate
FROM exam_summaries es
JOIN exam_sessions s ON s.id = es.session_id
WHERE es.exam_id = ?
  AND s.status = 'submitted'
```

#### 3.3 阅卷答案分页加载

```javascript
// backend/routes/grading.js
// 改为按大题维度分批加载，不一次性拉全场
router.get('/tasks/:taskId/answers', auth, async (req, res) => {
  const { majorQuestionId, page = 1, pageSize = 50 } = req.query;
  const answers = await GradingModel.getAnswersByTask(taskId, {
    majorQuestionId,
    page: parseInt(page),
    pageSize: parseInt(pageSize),
  });
  res.json(answers);
});
```

#### 3.4 监控视频最新分片改 SQL 窗口函数

```sql
-- backend/models/examVideoChunkModel.js → listLatestByExam
SELECT t.* FROM (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY session_id
      ORDER BY created_at DESC
    ) AS rn
  FROM exam_video_chunks
  WHERE exam_id = ?
) t
WHERE t.rn = 1
```

### 方案四：架构级优化（100 人+ 目标）

#### 4.1 PM2 Cluster 模式（利用多核）

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'exam-backend',
    script: 'backend/server.js',
    instances: 'max',       // 按 CPU 核数自动扩展
    exec_mode: 'cluster',
    max_memory_restart: '1G',
    env: { NODE_ENV: 'production' },
  }]
};
```

> ⚠️ 启用 cluster 后，WebSocket 需改用 **Redis Pub/Sub** 跨进程广播，截图缓存需改存共享磁盘或 Redis。

#### 4.2 读写分离（MySQL 主从）

```javascript
// backend/config/database.js
// 写操作（答案保存、会话更新）→ 主库
// 读操作（汇总统计、监控查询、阅卷）→ 只读副本
const writePool = createPool(masterConfig);
const readPool = createPool(replicaConfig);
```

#### 4.3 汇总生成走独立 Worker

```javascript
// 使用 worker_threads 或独立进程处理 CPU 密集型汇总计算
// 避免汇总计算阻塞 HTTP 事件循环
const { Worker } = require('worker_threads');

function generateSummaryAsync(sessionId) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./workers/summaryWorker.js', {
      workerData: { sessionId }
    });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}
```

```

  ☐ 批量答案 UPSERT（单条多行）
  ☐ WS 心跳聚合推送（5 秒一批）
  ☐ findOrCreate 改 INSERT IGNORE + SELECT（减少往返）

阶段三：中期优化（管理端大数据，1 个月）

阶段四：架构升级（100 人+ 目标）
```

## 六、各阶段并发承载能力预期

| 优化阶段 | 可稳定支撑并发 | 主要瓶颈 |
|---------|-------------|---------|
| 当前现状 | **20–35 人**（开监控时） | 截图缓存溢出、N+1 开考、限流 |
| 完成阶段一 | **40–55 人** | 交卷高峰汇总并发、试卷 N+1 |
| 完成阶段二 | **80–120 人** | 单进程 CPU 上限 |
| 完成阶段三 | **120–200 人** | 管理端查询与考生端争连接 |
| 完成阶段四 | **300 人+** | 数据库写入成为瓶颈 |
