# 注册页并发优化方案

> 目标：保障 `/user/register.html` 在 **2000 人同时注册**场景下稳定、不卡顿。  
> 适用版本：当前 Node.js v22 + Express 4 + MySQL 8 单机架构。

---

## 一、瓶颈总览

| 编号 | 位置 | 类型 | 严重程度 |
|------|------|------|---------|
| B1 | `UserModel.create` — bcrypt cost=10 | CPU 密集，堵塞事件循环 | 🔴 最高 |
| B2 | `handleSelfRegister` — 6–8 次串行 DB 查询 | 连接池饥饿 | 🔴 最高 |
| B3 | 连接池默认 50，队列上限 500 | 高并发溢出 | 🔴 最高 |
| B4 | 验证码存进程内 `Map` | 多进程部署时跨进程失效 | 🟠 高 |
| B5 | 单进程 `node server.js` | CPU 吞吐上限 1 核 | 🟠 高 |
| B6 | `qms_users` 缺少唯一索引（可能） | 查询随数据量增长变慢 | 🟡 中 |
| B7 | 注册完成后多余 `findById` 查询 | 增加 1 次无效 DB 往返 | 🟡 中 |
| B8 | 前端从外部 CDN 加载 Tailwind / FA | 首屏渲染阻塞 500ms+ | 🟡 中 |

---

## 二、优化步骤（可按顺序逐步落地）

---

### Step 1：降低 bcrypt cost（立竿见影，5 分钟）

**文件**：`backend/models/userModel.js`，第 359 行

```js
// 改前
const passwordHash = await bcrypt.hash(password, 10);

// 改后（注册时用 8，登录验证不受影响）
const passwordHash = await bcrypt.hash(password, 8);
```

**效果**：bcrypt 哈希时间从 ~200ms 降至 ~50ms，libuv 线程池（4 线程）吞吐从 ~20 req/s 升至 ~80 req/s。注册场景已有手机号短信验证，安全性足够。

---

### Step 2：合并唯一性检查 + 删除冗余查询（减少 3 次 DB 往返）

**文件**：`backend/routes/auth.js`，`handleSelfRegister` 函数（约第 1576 行）

将原来的三次单独查询（查用户名 / 查手机号 / 查邮箱）合并为一条 SQL，并在注册成功后直接用已知数据构造响应，不再多查一次 `findById`。

```js
async function handleSelfRegister(req, res) {
  try {
    const body = req.body || {};
    const username = String(body.username || '').trim();
    const email = String(body.email || '').trim();
    const phoneRaw = String(body.phone || '').trim();
    const password = body.password != null ? String(body.password) : '';
    const userTypeRaw = String(body.userType || 'jobseeker').toLowerCase();
    const isEnterprise = userTypeRaw === 'enterprise' || userTypeRaw === 'company';

    // —— 基础校验（不变）——
    if (!username || username.length < 2)
      return res.status(400).json({ success: false, message: '用户名至少 2 个字符' });
    if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email))
      return res.status(400).json({ success: false, message: '请填写有效邮箱' });
    const phoneDigits = phoneRaw.replace(/\D/g, '');
    if (phoneDigits.length < 10)
      return res.status(400).json({ success: false, message: '请填写有效手机号' });
    if (!password || password.length < 6)
      return res.status(400).json({ success: false, message: '密码至少 6 位' });

    // —— 【改】三合一唯一性检查，单次查询替换原来三次 ——
    const phoneNorm = phoneDigits;
    const portalRole = isEnterprise ? 'enterprise' : 'user';
    const [dupRows] = await pool.execute(
      `SELECT username, phone, email, role FROM qms_users
       WHERE username = ? OR email = ? OR phone = ?
       LIMIT 5`,
      [username, email, phoneNorm]
    );
    for (const row of dupRows || []) {
      if (row.username === username)
        return res.status(400).json({ success: false, message: '用户名已被使用' });
      if (row.email === email)
        return res.status(400).json({ success: false, message: '邮箱已被注册' });
      const rowRole = normalizeRoleForPortal(row.role);
      if (row.phone === phoneNorm) {
        const roleMatch = isEnterprise ? rowRole === 'enterprise' : (rowRole !== 'enterprise' && rowRole !== 'admin');
        if (roleMatch)
          return res.status(400).json({
            success: false,
            message: isEnterprise
              ? '该手机号已注册企业账号，请直接「政企登录」'
              : '该手机号已注册求职者账号，请直接登录个人中心'
          });
      }
    }

    // —— 短信验证码校验（不变）——
    if (aliyunSms.isConfigured()) {
      const smsCode = String(body.smsCode || body.code || '').trim();
      const vr = verifyAndConsumeSmsCode(phoneDigits, smsCode);
      if (!vr.ok)
        return res.status(400).json({ success: false, message: vr.message });
    }

    // —— 创建用户（不变）——
    const role = isEnterprise ? 'enterprise' : 'user';
    const userId = await UserModel.create({
      username, password, role, email,
      real_name: username,
      phone: phoneNorm
    });
    await UserModel.updatePhoneSafe(userId, phoneNorm);

    // —— 企业档案（不变）——
    let enterpriseId = null;
    if (isEnterprise) {
      try {
        enterpriseId = await EnterpriseModel.create({
          name: username, contactName: username,
          contactPhone: phoneNorm, contactEmail: email,
          address: null, userId
        });
      } catch (ee) {
        console.error('register-self enterprise:', ee.message);
        await pool.execute('DELETE FROM qms_users WHERE id = ?', [userId]).catch(() => {});
        return res.status(500).json({ success: false, message: '企业档案创建失败，请稍后重试' });
      }
    }

    // —— 【改】直接构造响应，不再多查一次 findById ——
    const token = generateToken(
      { id: userId, username, role },
      { portal: isEnterprise ? 'enterprise' : 'jobseeker' }
    );

    return res.status(201).json({
      success: true,
      data: {
        userId, username, email,
        userType: isEnterprise ? 'enterprise' : 'jobseeker',
        accessToken: token, token, enterpriseId
      }
    });
  } catch (error) {
    console.error('register-self error:', error);
    if (error && error.code === 'ER_DUP_ENTRY')
      return res.status(400).json({ success: false, message: '用户名、手机或邮箱已存在' });
    return res.status(500).json({ success: false, message: '注册失败：' + (error.message || '服务器错误') });
  }
}
```

**效果**：注册 DB 往返从 6–8 次降至 **3–4 次**（唯一性检查 1 次 + create 1 次 + updatePhone 1 次）。

### Step 3：扩大数据库连接池

**文件**：`backend/.env`

```bash
# 原值
DB_POOL_SIZE=50

# 改为（根据服务器内存调整，建议 100–150）
DB_POOL_SIZE=100
DB_POOL_QUEUE_LIMIT=2000
```

同时在 MySQL 侧执行（需 root 权限）：

```sql
-- 查看当前值
SHOW VARIABLES LIKE 'max_connections';

-- 临时修改（重启后失效）
SET GLOBAL max_connections = 300;

-- 永久修改：在 my.cnf / my.ini 的 [mysqld] 段添加
-- max_connections = 300
```

**效果**：消除高并发时的 `pool queue overflow` 错误；100 连接在 MySQL 内存上额外占用约 50–100MB，可接受。

### Step 4：Redis 化验证码存储

**背景**：下一步将启用 PM2 多进程，进程内 `Map` 在多进程下无法共享，必须先做此步。

#### 4.1 安装依赖

```bash
cd backend
npm install ioredis
```

并在 `backend/.env` 添加：

```bash
REDIS_URL=redis://127.0.0.1:6379
# 若 Redis 有密码：redis://:yourpassword@127.0.0.1:6379
```

#### 4.2 创建 Redis 客户端模块

新建文件 `backend/services/redisClient.js`：

```js
const Redis = require('ioredis');

let client = null;

function getRedisClient() {
  if (client) return client;
  const url = process.env.REDIS_URL;
  if (!url) return null; // 未配置时回退到内存 Map
  client = new Redis(url, {
    lazyConnect: true,
    maxRetriesPerRequest: 2,
    enableOfflineQueue: false,
  });
  client.on('error', (e) => console.warn('[Redis]', e.message));
  return client;
}

module.exports = { getRedisClient };
```

#### 4.3 替换 `routes/auth.js` 中的验证码存取逻辑

找到 `routes/auth.js` 中的如下代码段（约第 608–634 行），整体替换：

```js
// ——————————————————————————————————————————
// 短信验证码存取（优先 Redis，降级内存 Map）
// ——————————————————————————————————————————
const { getRedisClient } = require('../services/redisClient');
const SMS_EXPIRE_S = 5 * 60; // 5 分钟，Redis 用秒

// 内存降级备份（Redis 不可用时使用）
const smsCodeStore = new Map();
const SMS_EXPIRE_MS = SMS_EXPIRE_S * 1000;

function getSmsCode(phone) {
  const entry = smsCodeStore.get(phone);
  if (!entry || Date.now() > entry.expire) return null;
  return entry.code;
}
function setSmsCode(phone, code) {
  smsCodeStore.set(phone, { code, expire: Date.now() + SMS_EXPIRE_MS });
}

async function getSmsCodeAsync(phone) {
  const redis = getRedisClient();
  if (redis) {
    try { return await redis.get(`sms:${phone}`); } catch (_) {}
  }
  return getSmsCode(phone);
}
async function setSmsCodeAsync(phone, code) {
  const redis = getRedisClient();
  if (redis) {
    try {
      await redis.set(`sms:${phone}`, code, 'EX', SMS_EXPIRE_S);
      return;
    } catch (_) {}
  }
  setSmsCode(phone, code);
}
async function delSmsCodeAsync(phone) {
  const redis = getRedisClient();
  if (redis) {
    try { await redis.del(`sms:${phone}`); } catch (_) {}
  }
  smsCodeStore.delete(phone);
}
```

然后将 `send-sms` 路由里所有 `setSmsCode(norm, code)` / `setSmsCode(p, code)` 改为 `await setSmsCodeAsync(norm, code)` / `await setSmsCodeAsync(p, code)`，将 `verifyAndConsumeSmsCode` 函数改为 async 版本，内部查询改用 `await getSmsCodeAsync(...)` + `await delSmsCodeAsync(...)`。

### Step 5：PM2 多进程部署

**前置条件**：Step 4（Redis 化）已完成。

#### 5.1 安装 PM2

```bash
npm install -g pm2
```

#### 5.2 创建 PM2 配置文件

新建 `backend/ecosystem.config.js`：

```js
module.exports = {
  apps: [
    {
      name: 'sjrcw-backend',
      script: 'server.js',
      cwd: __dirname,
      instances: 'max',        // 按 CPU 核数自动扩展
      exec_mode: 'cluster',    // 多进程负载均衡
      max_memory_restart: '1G',
      env: {
        NODE_ENV: 'production',
        PORT: 3000
      },
      // 优雅重启：新进程就绪后再关旧进程
      wait_ready: true,
      listen_timeout: 10000,
      kill_timeout: 5000,
      // 日志
      out_file: './logs/pm2-out.log',
      error_file: './logs/pm2-error.log',
      merge_logs: true,
      time: true
    }
  ]
};
```

#### 5.3 启动 / 重启

```bash
cd backend
mkdir -p logs
pm2 start ecosystem.config.js
pm2 save
pm2 startup  # 设置开机自启（按提示执行输出的命令）

# 查看状态
pm2 status
pm2 logs sjrcw-backend --lines 50
```

**效果**：4 核服务器 → 4 个 Node.js 进程，bcrypt 有效线程数从 4 升至 16，理论注册吞吐约 300+ req/s。

### Step 6：确认并补全数据库索引

```sql
-- 确认现有索引
SHOW INDEX FROM qms_users;

-- 若 username / phone / email 缺少唯一索引，补全
ALTER TABLE qms_users
  ADD UNIQUE INDEX IF NOT EXISTS idx_username (username),
  ADD UNIQUE INDEX IF NOT EXISTS idx_email    (email),
  ADD        INDEX IF NOT EXISTS idx_phone    (phone);
-- 注：phone 用普通索引（同号可能有多行），username/email 用唯一索引
```

> MySQL 8 支持 `IF NOT EXISTS`，不会影响已有索引。

### Step 7：前端静态资源本地化

**文件**：`legacy-shengju/user/register.html`（第 7–8 行）

```html
<!-- 改前（依赖外部 CDN，渲染阻塞） -->
<script src="https://cdn.tailwindcss.com"></script>
<link href="https://cdn.jsdelivr.net/npm/font-awesome@4.7.0/css/font-awesome.min.css" rel="stylesheet" crossorigin="anonymous">

<!-- 改后（本地路径，项目 libs/ 目录下已有 font-awesome） -->
<link href="/libs/font-awesome/css/font-awesome.min.css" rel="stylesheet">
<!-- Tailwind：用构建好的 CSS 文件替代运行时 CDN -->
<!-- 推荐：用 tailwindcss CLI 构建后放到 /libs/tailwind.min.css -->
<link href="/libs/tailwind.min.css" rel="stylesheet">
<!-- 删除 <script src="https://cdn.tailwindcss.com"></script> 这行 -->
```

构建 Tailwind CSS（一次性操作）：

```bash
# 如果全局没有 tailwindcss，先安装
npm install -g tailwindcss

# 仅提取 register.html 用到的样式（~8KB vs 全量 3MB）
tailwindcss -i /dev/null -o legacy-shengju/libs/tailwind-register.min.css \
  --content legacy-shengju/user/register.html --minify
```

**效果**：首屏加载减少 2 次外部 DNS + TCP 握手，节省约 500ms–1.5s 首屏时间（国内访问 tailwindcss.com CDN 有时较慢）。

## 四、各优化步骤的收益量化

| 步骤 | 改动量 | 耗时 | 注册吞吐提升 | 其他收益 |
|------|--------|------|-------------|---------|
| Step 1 bcrypt cost 8 | 1 行 | 5 min | 4× | 所有写密码操作均受益 |
| Step 2 合并查询 | ~50 行重写 | 1–2 h | 减少 50% DB 压力 | 注册延迟从 ~80ms→~30ms |
| Step 3 连接池扩容 | 2 行配置 | 10 min | 消除溢出错误 | 全站受益 |
| Step 4 Redis 验证码 | 新增 1 文件 + 修改 30 行 | 2–3 h | 多进程前置条件 | 重启不丢码 |
| Step 5 PM2 多进程 | 新增 1 配置文件 | 30 min | 线性×核数 | 全站吞吐翻倍 |
| Step 6 DB 索引 | 1 条 SQL | 5 min（在线 DDL） | 防止全表扫描 | 登录/搜索均受益 |
| Step 7 静态本地化 | 改 2 行 HTML | 30 min | 首屏 -500ms | 离线可用 |

## 五、压测验证建议

```bash

# 模拟 200 并发，持续 30 秒注册请求
autocannon -c 200 -d 30 -m POST \
```

## 六、Docker 环境补充（本地开发验证 Redis）

若本地用 Docker 运行服务，在 `deploy/local-docker/docker-compose.yml` 中添加 Redis 服务：

```yaml
services:
  # ... 已有的 mysql 服务 ...

  redis:
    image: redis:7-alpine
    container_name: sjrcw-redis
    ports:
      - "6379:6379"
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    restart: unless-stopped
```

然后 `.env` 中：

```bash
REDIS_URL=redis://redis:6379   # Docker 内部用服务名
```
